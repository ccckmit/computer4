# Index（資料庫索引）

## 概述

資料庫索引（Index）是一種特殊的資料結構，用於加速資料庫的查詢操作。索引類似於書籍的目錄：通過索引可以直接定位到目標資料，而不需要掃描整個資料表。

索引的主要代價是：
- **空間**：索引需要佔用額外的儲存空間
- **寫入效能**：每次寫入都需要更新索引

正確使用索引可以將查詢效能提升數百甚至數千倍。

## B-Tree 索引

B-Tree（及其變體 B+Tree）是最常見的索引類型：

```
                    [15 | 30]
                   /    |    \
             [<15]   [15-30]   [>30]
               │        │        │
             [..]      [..]     [..]
```

B-Tree 索引的特點：
- **有序**：鍵按順序排列，支持範圍查詢
- **平衡**：所有葉節點深度相同，查詢效能穩定
- **多分支**：高度低，減少 I/O 次數

## 索引的類型

### 單欄索引（Single Column Index）

```sql
CREATE INDEX idx_users_email ON users(email);
```

### 複合索引（Composite Index）

```sql
CREATE INDEX idx_orders_user_date ON orders(user_id, date);
```

**最左前綴原則**：查詢必須從索引的最左邊開始，才能有效使用複合索引。

```sql
-- 假設有複合索引 (user_id, date)
SELECT * FROM orders WHERE user_id = 1;        -- 使用索引
SELECT * FROM orders WHERE date = '2024-01-01';  -- 不使用索引
SELECT * FROM orders WHERE user_id = 1 AND date = '2024-01-01';  -- 使用索引
```

### 唯一索引（Unique Index）

```sql
CREATE UNIQUE INDEX idx_users_email ON users(email);
```

唯一索引確保索引列的值唯一。

### 主鍵索引（Primary Key Index）

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY,  -- 自動建立唯一索引
    name TEXT
);
```

主鍵索引是一種特殊的所有值唯一的索引。

### 全文索引（Full-Text Index）

```sql
-- MySQL
ALTER TABLE articles ADD FULLTEXT(title, content);

-- PostgreSQL
CREATE INDEX idx_articles_content ON articles USING gin(to_tsvector('english', content));
```

### 空間索引（Spatial Index）

用於地理資料型別（如 PostGIS）：

```sql
-- PostgreSQL with PostGIS
CREATE INDEX idx_locations_geo ON locations USING gist(geom);
```

## 索引的資料結構選擇

| 資料結構 | 適用場景 | 範例 |
|----------|----------|------|
| B-Tree | 一般用途、等值查詢、範圍查詢 | 大多數場景 |
| Hash | 僅等值查詢、不支持範圍 | 記憶體資料庫、字典 |
| GiST | 幾何資料、範圍、全文檢索 | PostGIS |
| GIN | 多值屬性、JSON、陣列 | 文件搜尋、JSON 欄位 |
| R-Tree | 空間資料 | 地理應用 |

## 索引的原理

### B-Tree 索引建構

假設有 100 萬筆記錄，每頁 100 條，需要多少層？

- 第 0 層（葉）：100 萬 / 100 = 10,000 頁
- 第 1 層：10,000 / 100 = 100 頁
- 第 2 層：100 / 100 = 1 頁

只需要 3 層！

### 查詢流程

```sql
SELECT * FROM users WHERE email = 'test@example.com';
```

1. 從根節點開始，在節點內二分搜尋
2. 找到包含目標鍵的指標
3. 下降到子節點
4. 重複直到到達葉節點
5. 在葉節點中找到目標記錄

### 覆蓋索引（Covering Index）

如果所需的資料都在索引中，不需要回表：

```sql
-- 假設有複合索引 (user_id, name)
SELECT user_id, name FROM users WHERE user_id = 1;
-- 這個查詢可以直接從索引返回結果，不需要訪問資料表
```

## 索引的最佳實踐

### 什麼時候建立索引

1. **WHERE 子句中常用的列**
2. **JOIN 的連接列**
3. **ORDER BY 的列**
4. **GROUP BY 的列**
5. **高基數（high cardinality）的列**

### 什麼時候不應該建立索引

1. **低基數的列**：如性別（只有兩個值）、狀態標誌
2. **很少被查詢的列**
3. **大文字欄位**：使用全文索引代替
4. **頻繁寫入的表**：每次寫入都要更新索引

### 索引命名規範

```sql
-- 通用格式
idx_{table}_{column(s)}

-- 範例
idx_orders_user_id
idx_orders_user_date
idx_users_email
```

### 監控索引使用

```sql
-- PostgreSQL
SELECT indexrelname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE relname = 'orders';

-- MySQL
SHOW INDEX FROM orders;
```

## 索引的問題

### 索引失效

以下情況會導致索引無法使用：

```sql
-- 使用函數
WHERE YEAR(created_at) = 2024
-- 應改為
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'

-- 使用 LIKE 前綴
WHERE name LIKE '%abc'
-- 如果是前綴匹配 LIKE 'abc%'，可以使用索引

-- 隱式類型轉換
WHERE user_id = '123'  -- user_id 是 INTEGER
-- 應改為
WHERE user_id = 123

-- 使用 OR 連接不同列的條件
WHERE user_id = 1 OR email = 'test@example.com'
-- 如果 email 列沒有索引，可能導致全表掃描
```

### 索引膨脹

隨著資料的增刪，索引可能變得稀疏：

```sql
-- PostgreSQL: 重建單一索引
REINDEX INDEX idx_users_email;

-- MySQL: 線上重建
ALTER TABLE users DROP INDEX idx_users_email, ADD INDEX idx_users_email(email);
```

## 在 db6 中的 Index

db6 目前專注於 KV 儲存，索引的概念通過以下方式體現：

```rust
// StorageEngine trait
fn scan(&self, table_id: u32, start: &[u8], end: &[u8]) -> Result<Vec<(Vec<u8>, Vec<u8>)>>;
```

在 KV 模型中，鍵的設計本身就是一種「索引策略」：
- 如果要按 `user_id` 查詢，可以將鍵設計為 `user:{id}:{data}`
- 如果要按 `email` 查詢，可以建立反向索引 `email:{value}:user`

未來 db6 可能會在 SQL 層之上增加自動索引管理。
