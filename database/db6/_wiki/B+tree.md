# B+Tree（B+ 樹）

## 概述

B+Tree（B+ 樹）是 B-Tree 的一種重要變體，在資料庫系統中佔據主導地位。與 B 樹不同，B+Tree 的所有資料都儲存在葉節點，內部節點只儲存鍵作為分割點。這種設計使得 B+Tree 能夠最大化分支因子，進一步降低樹的高度，同時為範圍查詢提供高效的支援。

## 與 B-Tree 的核心差異

B+Tree 相比 B-Tree 有三個關鍵設計差異：

### 1. 資料只存在葉節點

B-Tree 的任何節點都可以儲存資料，而 B+Tree 只有葉節點儲存實際的鍵值對。這意味著：

- 內部節點可以儲存更多的鍵（因為不需要附帶資料）
- 分支因子更大，樹的高度更低
- 查詢路徑都會到達葉節點，所有查詢時間相同

### 2. 葉節點之間有雙向鏈結

B+Tree 的葉節點通過雙向指標串連，形成一個有序的雙向鏈結串列：

```
[葉節點1] <-> [葉節點2] <-> [葉節點3] <-> ...
```

這個設計為範圍查詢提供了極大的便利：找到起始位置後，可以沿著鏈結順序走訪所有葉節點，無需回溯到父節點。

### 3. 葉節點包含衛星資料

在 B-Tree 中，內部節點的每個鍵旁邊可能附帶資料。而在 B+Tree 中，葉節點儲存所有鍵的完整資料（或指向資料的指標）。這使得：

- 內部節點更精簡
- 葉節點可以容納更多記錄
- 葉節點的掃描可以一次讀取連續的資料

## 結構示意圖

一棵典型的 B+Tree（m=3）：

```
                    [15 | 30]
                   /    |    \
         [<15]        [15-30]        [>30]
          |              |              |
       [葉節點1]     [葉節點2]       [葉節點3]
      [1,5,10]       [20,25]       [35,40]
         ^               ^               ^
         |               |               |
         +----<-><-><-><-><-><-><-><->+
```

## 查詢效能分析

假設：
- 頁大小 = 16KB
- 鍵大小 = 8 bytes
- 每頁可容納約 1000 個鍵

一棵 3 層的 B+Tree 可以儲存：
- 第 1 層：1 個根節點
- 第 2 層：最多 1000 個內部節點
- 第 3 層：最多 1000 × 1000 = 1,000,000 個葉節點
- 總容量：1,000,000 × (16KB / 8 bytes) ≈ 20 億條記錄

這就是為什麼 B+Tree 只需要 3 次 I/O 即可查詢數十億筆資料。

## 範圍查詢的優勢

假設要查詢鍵值在 [20, 35] 範圍內的所有記錄：

1. 從根節點找到第一個葉節點（包含 20）
2. 在葉節點中使用二分搜尋找到起始位置
3. 沿著葉節點的鏈結，順序走訪所有在範圍內的葉節點
4. 收集所有匹配的記錄

這種走訪方式只涉及葉節點的順序讀取，無需回溯到內部節點，是 B+Tree 最大的優勢之一。

## 插入與刪除

### 插入

B+Tree 的插入流程：

1. 找到應插入記錄的葉節點
2. 在葉節點中按順序插入新鍵值對
3. 如果葉節點未滿，插入完成
4. 如果葉節點已滿，分裂為兩個節點
5. 在父節點中添加新的分割鍵（複製葉節點的最小鍵）
6. 如果父節點也需要分裂，繼續向上傳播

注意：B+Tree 分割時，內部節點需要複製而不是移動鍵，這是與 B-Tree 的重要區別。

### 刪除

B+Tree 的刪除流程：

1. 找到包含目標鍵的葉節點
2. 刪除該鍵值對
3. 如果葉節點的鍵數量過少，需要進行「借用」或「合併」操作：
   - **借用**：從相鄰的葉節點借一個鍵過來
   - **合併**：與相鄰葉節點合併成一個
4. 如果發生合併，可能需要更新父節點的分割鍵
5. 合併操作可能向上傳播

## 叢集索引 vs 非叢集索引

B+Tree 在資料庫中有兩種主要用途：

### 叢集索引（Clustered Index）

- 葉節點儲存完整的資料列（而不是僅僅指向資料的指標）
- 一個表只有一個叢集索引（通常就是主鍵索引）
- 資料與索引一起儲存，查詢時無需額外 I/O

### 非叢集索引（Non-Clustered Index）

- 葉節點儲存鍵值和對應資料的指標（或主鍵）
- 一個表可以有多個非叢集索引
- 查詢時需要先通過索引找到指標，再通過指標讀取資料

## 緩衝池（Buffer Pool）的角色

現代資料庫的 B+Tree 效能高度依賴於緩衝池：

1. 熱門節點（根節點、內部節點）常駐記憶體
2. 葉節點按需載入記憶體
3. 修改先在記憶體進行，定期刷回磁碟
4. 使用 LRU、LRU-K 或其他策略管理緩衝池

緩衝池使得大多數查詢可以在記憶體中完成，顯著提升效能。

## 與 LSM Tree 的比較

| 特性 | B+Tree | LSM Tree |
|------|--------|----------|
| 寫入效能 | 較低（隨機寫入） | 極高（順序寫入） |
| 讀取效能 | 較高（通常 3-4 次 I/O） | 較低（可能需要檢查多層） |
| 空間放大 | 較低 | 較高 |
| 寫入放大 | 較低 | 嚴重 |
| 交易支援 | 完整 | 有限 |

詳細比較請參閱 [LSM-tree.md](LSM-tree.md)。

## 應用場景

B+Tree 是以下場景的首選：

- **OLTP 資料庫**：需要高效的點查詢和交易處理
- **需要範圍查詢**：如時間範圍查詢、字母順序查詢
- **需要強一致性**：B+Tree 的原地更新機制適合交易
- **讀取密集型應用**：如電子商務、內容管理系統

## 主要資料庫的採用

- **MySQL (InnoDB)**：B+Tree 作為叢集索引和非叢集索引
- **PostgreSQL**：B+Tree（從 9.2 版本開始，之前的版本使用 heap + index 的分離結構）
- **SQL Server**：B+Tree
- **Oracle**：B+Tree
- **SQLite**：B+Tree

## 在 db6 中的應用

db6 的 [BTreeEngine](../src/engine/btree/) 實現使用 B-Tree 結構。雖然名稱是 BTree，但實際實現融合了 B+Tree 的特點（葉節點串連用於範圍查詢）。這是因為 SQLite 本身的 B-Tree 就是一種 B+Tree 變體。
