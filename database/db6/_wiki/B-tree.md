# B-Tree（B 樹）

## 概述

B-Tree（B 樹）是一種自平衡的搜尋樹，1970 年由 Rudolf Bayer 和 Edward McCreight 在 Boeing 實驗室發明。B 樹的設計目標是減少磁碟存取次數，將树的分支因子最大化，使得樹的高度保持較低。B 樹廣泛應用於關聯式資料庫的索引結構，是現代資料庫系統最重要的資料結構之一。

## 設計動機

在了解 B 樹之前，需要理解磁碟存取的代價問題。與記憶體存取（通常小於 100 奈秒）相比，磁碟存取延遲通常在毫秒級，差了約 10 萬倍。因此，如果資料庫的索引結構需要存取磁碟多次（如二叉樹），效能會嚴重受限。

B 樹的核心思想是：**最大化每個節點的子節點數量**，將樹的高度降到最低。這樣一來，查詢任何資料都只需要極少的磁碟存取次數（通常 3-4 次即可覆蓋數十億筆資料）。

## 結構特性

B 樹的典型特性：

1. **每個節點最多有 m 個子節點**（m 稱為 B 樹的 order）
2. **每個內部節點至少有 ⌈m/2⌉ 個子節點**（根節點除外）
3. **所有葉節點在同一層**：樹是平衡的，確保搜尋路徑長度一致
4. **非葉節點有 n+1 個指標**：如果有 n 個鍵，就有 n+1 個子節點指標
5. **鍵是有序的**：每個節點內的鍵按順序排列，支持二分搜尋

一棵典型的 B 樹如下圖所示（m=3 的情況）：

```
        [15 | 30]
       /    |    \
  [<15]  [15-30]  [>30]
```

## B-Tree 的查找

B 樹的搜尋類似二叉搜尋樹，但每個節點內需要進行線性或二分搜尋：

1. 從根節點開始，在節點內搜尋目標鍵
2. 如果找到，返回該值
3. 如果沒找到，根據鍵的范圍選擇對應的子節點
4. 重複直到找到或達到葉節點

時間複雜度：O(log n)，實際磁碟存取次數約為 O(log_{m/2}(n))

## B-Tree 的插入

B 樹的插入需要維持平衡性質：

1. 找到目標葉節點
2. 在葉節點中按順序插入新鍵
3. 如果葉節點未滿，插入完成
4. 如果葉節點已滿，分裂成兩個節點，並將中間鍵提升到父節點
5. 如果父節點也需要分裂，繼續向上傳播

## B-Tree 的刪除

刪除操作較為複雜，需要處理兩種情況：

1. **刪除葉節點中的鍵**：直接刪除，如果節點鍵數量過少（低於 ⌈m/2⌉-1），需要從兄弟節點借鍵或合併節點
2. **刪除內部節點中的鍵**：用前驅或後繼鍵替換，然後刪除葉節點中的替換鍵

合併操作可能導致父節點也需要合併，稱為「級聯合併」。

## 分支因子的重要性

B 樹的分支因子取決於：
- 節點大小（通常為頁大小，如 4KB 或 8KB）
- 鍵和指標的大小

假設：
- 頁大小 = 4KB
- 鍵大小 = 8 bytes
- 指針大小 = 8 bytes
- 每個節點可容納 (4KB) / (8+8) ≈ 256 個鍵

則：
- 分支因子 ≈ 257
- 高度為 2 時，可儲存 257² ≈ 66,000 鍵
- 高度為 3 時，可儲存 17,000,000 鍵

這就是為什麼 B 樹只需要 3-4 次磁碟存取即可查詢數十億筆資料。

## 與二叉搜尋樹的比較

| 特性 | B-Tree | 二叉搜尋樹 |
|------|--------|-----------|
| 分支因子 | 高（數百） | 2 |
| 高度 | 低（3-4 層） | 高（數十層） |
| 磁碟存取 | 少（3-4 次） | 多（數十次） |
| 記憶體效率 | 低（節點未滿會浪費空間） | 高 |
| 實現複雜度 | 中等 | 簡單 |

## 變體：B+Tree

B+Tree 是 B 樹的最重要變體，在資料庫系統中幾乎取代了原始 B 樹。B+Tree 的主要改進：

1. **所有資料存在葉節點**：內部節點只存鍵，不存資料
2. **葉節點串連**：雙向鏈結串列，優化範圍查詢
3. **更多的分支因子**：因為內部節點不需要儲存資料

詳細說明請參閱 [B+tree.md](B+tree.md)。

## 應用場景

B 樹及其變體幾乎應用於所有關聯式資料庫：

- MySQL 的 InnoDB 儲存引擎使用 B+Tree
- PostgreSQL 使用 B-Tree（及時更新版本也用 B+Tree）
- SQLite 使用 B-Tree
- Oracle 使用 B-Tree

此外，許多 NoSQL 資料庫的索引也基於 B 樹或其變體。

## 優點

- **高效的範圍查詢**：葉節點有序，支持範圍掃描
- **適合磁碟儲存**：高度低，減少磁碟存取
- **自平衡**：自動維持平衡，效能穩定
- **簡單**：實現相對簡單，穩定性高

## 缺點

- **寫入效能較低**：每次插入/刪除可能需要頁分裂或合併，涉及磁碟隨機寫入
- **空間放大**：節點未滿時會浪費空間
- **不適合純寫入場景**： LSM Tree 在純寫入場景效能更優

## 在 db6 中的應用

db6 專案的 [BTreeEngine](../src/engine/btree/) 基於移植自 sql6 的 pager/btree 實現。B 樹是本專案三個引擎中最成熟、功能最完整的實現，支援完整的交易機制（ACID）。
