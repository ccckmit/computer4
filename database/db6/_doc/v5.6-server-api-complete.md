# v5.6 Server API Complete

## Overview

All three server transports (HTTP, WebSocket, UDS) now provide complete API coverage for KV, SQL, FTS, and Msgq operations.

| POST | `/kv/delete` | Delete key |
| POST | `/kv/scan` | Range scan |
| POST | `/kv/batch_put` | Batch write |
| POST | `/kv/range_delete` | Range delete |
| GET | `/kv/stats` | Engine statistics |
| POST | `/sql/execute` | Execute SQL |
| POST | `/fts/insert` | Insert FTS document |
| POST | `/fts/search` | Full-text search |
| POST | `/fts/search_bm25` | BM25 search |
| POST | `/queue/create` | Create queue |
| POST | `/queue/enqueue` | Enqueue message |
| POST | `/queue/dequeue` | Dequeue message |
| POST | `/queue/ack` | Ack message |
| POST | `/queue/stats` | Queue statistics |

| POST | `/pubsub/subscribe` | Subscribe to channel |
| GET | `/ws` | WebSocket upgrade |

### WebSocket / UDS (Unix Socket)

Same unified JSON-RPC protocol over WebSocket binary/text frames and Unix socket.

| Method | Params | Description |
|--------|--------|-------------|
| `health` | - | Health check |
| `kv_put` | table_id, key, value | Put key-value |
| `kv_get` | table_id, key | Get value |
| `kv_delete` | table_id, key | Delete key |
| `kv_scan` | table_id, start, end | Range scan |
| `kv_batch_put` | table_id, pairs | Batch write |
| `kv_range_delete` | table_id, start, end | Range delete |
| `kv_stats` | - | Engine stats |
| `sql_execute` | sql | Execute SQL |
| `fts_insert` | doc_id, text | Insert FTS doc |
| `fts_search` | query | Full-text search |
| `fts_search_bm25` | query | BM25 search |
| `queue_create` | name | Create queue |
| `queue_enqueue` | name, payload, timeout | Enqueue message |
| `queue_dequeue` | name, wait_timeout | Dequeue message |
| `queue_ack` | name, msg_id | Ack message |
| `queue_stats` | name | Queue stats |
| `queue_list` | - | List all queues |
| `pubsub_publish` | channel, payload | Publish message |
| `pubsub_subscribe` | channel | Subscribe to channel |

## Request/Response Format

### HTTP

```bash
# KV
curl -X POST http://localhost:50052/kv/put \
  -H "Content-Type: application/json" \
  -d '{"table_id":1,"key":"user:1","value":"{\"name\":\"John\"}"}'

# SQL
curl -X POST http://localhost:50052/sql/execute \
  -H "Content-Type: application/json" \
  -d '{"sql":"SELECT * FROM users"}'

# FTS
curl -X POST http://localhost:50052/fts/search \
  -H "Content-Type: application/json" \
  -d '{"query":"database"}'

# Queue
curl -X POST http://localhost:50052/queue/enqueue \
  -H "Content-Type: application/json" \
  -d '{"name":"myqueue","payload":"hello","timeout":30}'
```

### WebSocket / UDS

```json
// Request
{"method": "kv_put", "id": 1, "params": {"table_id": 1, "key": "test", "value": "hello"}}

// Response
{"id": 1, "result": {"status": "ok"}, "error": null}

// Request
{"method": "fts_search", "id": 2, "params": {"query": "database"}}

// Response
{"id": 2, "result": {"doc_ids": [1, 5, 9]}, "error": null}
```

## Implementation Notes

### FTS

FTS index is created from the shared KvEngine. Index data is stored in table_id=255.

### Msgq

Msgq is created from the shared KvEngine. Queue data uses prefixed keys:
- `queue:{name}:data:{id}` - message payload
- `queue:{name}:meta` - queue metadata

### SQL

SQL executor uses its own in-memory engine (separate from the main KV engine). This is intentional for isolation.

## Dependencies

No new dependencies required. All features use existing crates:
- `tokio` with `net` feature for UDS
- `serde_json` for JSON handling
- `axum` for HTTP/WebSocket
