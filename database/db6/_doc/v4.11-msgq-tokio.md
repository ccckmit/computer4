# v4.11 - Msgq 模組深度重構與 Tokio 效能優化

**狀態：`已完成`**

## 完成項目

在經歷 v4.10 的轉換陣痛期後，我們發現了 `sync_queue` 與 `async_queue` 在面對 `crate::error::Error` (底層 DB Error) 的轉換不相容挑戰。本版目標從**底層 Error mapping** 出發，徹底解決了不相容的問題。接著進一步重組了 `msgq` 的公用模組，以期最大程度地釋放 **Tokio** 非同步執行時的資源優勢。

### 1. 錯誤處理相容確認
* 經過測試驗證，`msgq/error.rs` 裡的 `Db(#[from] DbError)` 已能自動相容 `crate::error::Error`。所有 `cargo test` 與 `cargo check` 確保穩定通過，不需要進一步重寫 Error mapping。

### 2. 目錄與邏輯拆分：`AsyncQueue`
* **問題**：原本的 `AsyncQueue` 超過 700 行且僅為 In-Memory（記憶體內）實作，完全沒有寫入 `StorageEngine`，造成嚴重分歧且與 `SyncQueue` 架構分離。
* **重構解法**：
    * 創建新的 `src/msgq/async_queue/` 模組，並將業務邏輯拆分至 `config.rs`, `metrics.rs`, `queue.rs`, `stream.rs`, `exactly.rs`, `facade.rs` 等獨立檔案中。
    * 實踐 **完美的 DRY 原則**：藉由更改 `AsyncQueue` 結構直接包含 `SyncQueue` 引擎 (`inner: Arc<RwLock<SyncQueue>>`)，將核心資料庫資料落地、存放、存取的操作直接下放交由 `SyncQueue` 執行。
    * **Tokio `Notify` 取代阻塞**：原本 `dequeue()` 使用極度不效率的 `tokio::time::sleep(1)`，現已全面重構為 `tokio::sync::Notify` 喚醒機制。在 `enqueue` 將資料推進資料庫時觸發 Notify，結合 `tokio::select!`，實現真正的非阻塞、零次詢問、低延遲消息等候。
    * **型別別名統一**：消除重複的 `AsyncQueueMessage`，實作了 alias 直接指向 `SyncQueueMessage`。

### 3. 架構同步：`AsyncPubSub`
* **重構解法**：同樣重構整個 `src/msgq/async_pubsub.rs` 檔案，改為透過包裝 `SyncPubSub` 實現資料庫持久化。
* **Tokio `broadcast` 機制融合**：將 DB 寫入與 `tokio::sync::broadcast` 推播完美整合。每次 `publish` 訊息會先寫入資料庫（支援 `subscribe_with_history()` 找回過去紀錄），然後直接利用 `broadcast` 發送給目前在線的異步監聽器。這兼顧了無資料庫查詢壓力的極低延遲，並保留了完整的意外回滾與備份能力。
* **Msgq API 擴充**：在 `src/msgq/mod.rs` 主介面新增 `pub fn async_pubsub(&self)` 方法，讓初始化 API 一把抓，操作完整對齊 `async_queue`。

## 結論與測試

本次重構牽涉了數百行的邏輯置換與模組重構，統一了 Sync / Async 版本的底層資料模型，同時維持 Async 版本強大的 Tokio 併發支援。經測試，全部共 `166 passed` 的核心測試均已順利過關！成功發揮了非同步運算的效能，標誌著 `msgq` 模組正式具備可用的極高品質基礎架構。
