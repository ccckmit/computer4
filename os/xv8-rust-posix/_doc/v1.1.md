# v1.1 — alloc / libposix Dual Target

> 主題：讓 xv8 user space 完整支援 Rust `alloc` crate，使 `Vec`/`String` 可用，並規劃 `os/posix/tools` 不需複製即可在 xv8 上編譯執行。


---

## 1. 背景：Linux 的軟體堆疊 vs xv8

Linux 上 Rust `std` 並非憑空出現，層層依賴：

```
┌─────────────────────────────────────┐
│ Rust std (Vec, String, Box, etc.)  │  ← 開發者直接使用
├─────────────────────────────────────┤
│ alloc crate                         │  ← #[global_allocator]
├─────────────────────────────────────┤
│ malloc (libc)                       │  ← 需要 brk/sbrk/mmap
├─────────────────────────────────────┤
│ Linux Kernel syscalls               │  ← brk(2), mmap(2)
├─────────────────────────────────────┤
│ Linux Kernel (實體記憶體)            │
└─────────────────────────────────────┘
```

xv8 目標：模仿 Linux 提供給 `alloc` 的底層介面，讓 `Vec`/`String` 在 xv8 上自動可用。

---

## 2. xv8 現況分析

### 2.1 Memory Model（已確認的設計）

xv8 是 **static binary**（kernel + user 靜態連結為單一 ELF）。Rust 的 `#[global_allocator]` 是 crate-level，只能有一個。

| 元件 | 職責 |
|------|------|
| `kernel/src/kalloc.rs` | 宣告 `#[global_allocator]`，使用 `buddy-alloc` 管理實體記憶體頁面池 |
| `kernel/src/sysproc.rs:117` (`sys_sbrk`) | syscall 12，向 buddy-alloc 請求 n bytes，回傳指標 |
| `user/src/lib.rs` | `#![no_std]` + `#![feature(alloc)]` + `extern crate alloc` |
| `user/src/args.rs` | `Args::from_stack()` 從 stack 讀取 argc/argv |

**共用方式**：user 不宣告自己的 `#[global_allocator]`，直接使用 kernel 宣告的那個。`alloc` crate 的 `GlobalAlloc::alloc()` 內部呼叫 buddy-alloc，需要時透過 `sbrk(12)` 或 `mmap(34)` syscall 向 kernel 要記憶體。

### 2.2 現有問題

| 問題 | 說明 |
|------|------|
| `libposix::args()` on xv8 | 目前回 `Vec::new()`（placeholder），需要實作從 `Args` → `Vec<String>` |
| `libposix` 尚未支援 xv8 | `os/posix/libposix/` 只針對 Mac (`#[cfg(unix)]`)，缺少 xv8 (`#[cfg(target_os = "none")]`) 的 syscall wrappers |
| `os/posix/tools` 無法直接引用 | 工具使用 `std::fs::File`、`std::env::args()` 等，需改用 `libposix` 包裝 |
| `File::read_to_end()` 缺乏 | `Read` trait 的 `read` 會 block，utility function `read_to_end` 尚未實作 |

## 3. 實作規劃

### Phase 1：讓 `Vec<String>` 在 xv8 上運作

**目標**：`libposix::args()` 在 xv8 上回傳有意義的 `Vec<String>`。

#### Step 1.1：實作 `libposix::args()` on xv8

`libposix/io.rs` 在 `#[cfg(target_os = "none")]` 區塊：

```rust
pub fn args() -> Vec<String> {
    let args = Args::from_stack();
    args.args_as_str()  // Iterator<Item = &str>
        .map(|s| String::from(s))
        .collect()
}
```

需要 `extern crate alloc` 已經在 `user/src/lib.rs` 引入。

**驗證**：`cat` 或 `echo` 工具輸出 `args()` 確認內容正確。

#### Step 1.2：加 `read_to_end` helper（可選，延後）

```rust
pub fn read_to_end<R: Read>(r: &mut R, buf: &mut Vec<u8>) -> Result<usize>
```

用 loop + `Read::read()` 實作。需 `alloc::vec::Vec`（已支援）。

---

### Phase 2：`libposix` 兩路實作（核心工作）

**目標**：`libposix/src/io.rs` 同時支援 Mac 和 xv8，提供統一的 `File`/`Read`/`Write`/`args`/`print`/`println`/`exit` API。

#### Step 2.1：確認現有 `libposix` 結構

`libposix/io.rs` 目前內容：
- `Read` / `Write` traits（Unix 實作）
- `File` struct with `read`/`write`/`close`
- `Stdin` / `Stdout` / `Stderr` singletons
- `sys` module（`#[cfg(target_os = "none")]`）：`open`/`read`/`write`/`close` inline asm
- `print_impl`（Unix 和 xv8 各一組）

**需要修改**：
- `sys` module 加 `exit` (ecall a7=10)
- `print_impl` on xv8：`println` 用 `sys_write` 而非 `print!` macro
- `lib.rs` 匯出 `exit` 和 `args`

#### Step 2.2：改造 `od.rs` 作為示範

`os/posix/tools/src/bin/od.rs` 目前使用：
```rust
use std::fs::File;
use std::io::Read;
use std::env::args;
use std::process::exit;
```

改為：
```rust
use libposix::{File, Read, args, exit, print, println};
```

**改動點**：
- `std::fs::File` → `libposix::File`
- `std::env::args()` → `libposix::args()`
- `std::process::exit(1)` → `libposix::exit(1)`
- `print!()` / `println!()` → `libposix::print()` / `libposix::println()`
- `File::open(Path::new(&files[0]))` → `File::open(&files[0])`（`libposix::File` 直接接受 `&str`）
- `std::io::stdin().lock().read_to_end(&mut data)` → 需要 `read_to_end` 或自己 loop

#### Step 2.3：同樣改造 `cat.rs`、`head.rs`

流程與 `od.rs` 相同。

---

### Phase 3：xv8 user 引用 `libposix`

**目標**：`xv8-rust-posix/user` 直接 `use libposix::...`，不複製 source code。

`xv8-rust-posix/user/Cargo.toml` 加：
```toml
[dependencies]
libposix = { path = "../../posix/libposix" }
```

`user/src/lib.rs` 加上 `pub use libposix::*` 或選擇性 re-export。

**注意**：xv8 的 `Cargo.toml` 是 2024 edition，與 `libposix` 的 2021 edition 需確認 edition 差異是否造成問題（通常 Rust edition 可以混用）。

## 4. Syscall 需求

| 編號 | 名稱 | 用途 | 現況 |
|------|------|------|------|
| 12 | `Sbrk` | 調整 heap，給 alloc 用 | ✅ 已實作 (`sys_sbrk`) |
| 34 | `Mmap` | 匿名映射，給 alloc 大塊配置 | ✅ 已實作 (`sys_mmap`) |
| 5 | `Read` | 讀取檔案/stdin | ✅ 已實作 |
| 16 | `Write` | 寫入 stdout/stderr | ✅ 已實作 |
| 15 | `Open` | 開啟檔案 | ✅ 已實作 |
| 21 | `Close` | 關閉檔案描述符 | ✅ 已實作 |
| 10 | `Exit` | 終止行程 | ✅ 已實作 |

所需 syscall 皆已存在，無需新增。
