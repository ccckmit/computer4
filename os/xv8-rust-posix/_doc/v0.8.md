# v0.8 — 權限模型 (Permission Model)

> 新增使用者/群組 ID、檔案權限模式、umask，以及相關系統呼叫。新建立檔案會繼承行程的 uid/gid。

## 任務

- [x] **`sys_getuid` / `sys_getgid`** — 回傳行程使用者/群組 ID
- [x] **`sys_setuid` / `sys_setgid`** — 設定行程使用者/群組 ID
- [x] **`sys_chmod`** — 變更檔案權限模式（保留型別位元）
- [x] **`sys_chown`** — 變更檔案擁有者（支援 `u32::MAX` 略過）
- [x] **`sys_umask`** — 檔案建立遮罩
- [x] **`ProcInner` 新增 `uid` / `gid`** — 行程憑證
- [x] **`ProcData` 新增 `umask`** — 預設 `0o022`
- [x] **`InodeInner` 新增 `uid` / `gid` / `mode`** — 檔案中繼資料
- [x] **建立時繼承** — `sys_open` / `sys_mkdir` / `sys_mknod` 設定 uid/gid + mode & !umask
- [x] **`fork()` 繼承** — 子行程複製父行程的 uid/gid/umask
- [x] **`mode` 模組** — `S_IFMT` / `S_IFREG` / `S_IFDIR` / `S_IFCHR` 常數
- [x] **`_posix_perm` 測試** — 6 項子測試

### mode 常數模組

```rust
pub mod mode {
    pub const S_IFMT: u16   = 0o170000;
    pub const S_IFDIR: u16  = 0o040000;
    pub const S_IFCHR: u16  = 0o020000;
    pub const S_IFBLK: u16  = 0o060000;
    pub const S_IFREG: u16  = 0o100000;
    pub const S_IFIFO: u16  = 0o010000;
    pub const S_IFLNK: u16  = 0o120000;
    pub const S_IFSOCK: u16 = 0o140000;

    pub fn from_type(r#type: InodeType) -> u16 {
        match r#type {
            InodeType::Directory => S_IFDIR | 0o755,
            InodeType::File     => S_IFREG | 0o644,
            InodeType::Device   => S_IFCHR | 0o666,
            InodeType::Free     => 0,
        }
    }
}
```

### 建立時權限繼承

在 `sys_open`、`sys_mkdir`、`sys_mknod` 中（`kernel/src/sysfile.rs`）：

```rust
let p_inner = proc.inner.lock();
inode_inner.uid = p_inner.uid;
inode_inner.gid = p_inner.gid;
drop(p_inner);
inode_inner.mode = mode::from_type(type) & !(data.umask as u16);
```

### fork() 繼承

```rust
new_inner.uid = parent_inner.uid;
new_inner.gid = parent_inner.gid;
// ...
new_data.umask = data.umask;
```

### sys_chmod — 保留型別位元

```rust
inner.mode = (inner.mode & 0o170000) | (mode & 0o7777) as u16;
```

### sys_chown — 可略過 uid/gid

```rust
if uid != u32::MAX { inner.uid = uid; }
if gid != u32::MAX { inner.gid = gid; }
```

### 使用者包裝層（`user/src/syscall.rs`）

| 函式 | 原始 | 安全 |
|------|------|------|
| getuid / getgid | `raw::getuid()` / `raw::getgid()` | `getuid() -> u32` / `getgid() -> u32` |
| setuid / setgid | `raw::setuid(uid)` / `raw::setgid(gid)` | `setuid(u32) -> Result` / `setgid(u32) -> Result` |
| chmod | `raw::chmod(path, mode)` | `chmod(&str, u16) -> Result` |
| chown | `raw::chown(path, uid, gid)` | `chown(&str, u32, u32) -> Result` |
| umask | `raw::umask(mask)` | `umask(u16) -> u16` |

## 修正的錯誤

### `Inode::get()` 重設 valid 導致 uid/gid 遺失

當 inode 的 refcount 降為 0（經由 `put()`）後重新查詢時，`Inode::get()` 會匹配到空閒插槽並設 `valid = false`，即使該插槽原本就含有相同的 `(dev, inum)`。這導致 `lock()` 重新從磁碟讀取，覆寫掉未持久化的 uid/gid（設為 0）。

**修正**：將匹配條件從 `ref > 0 && dev == dev && inum == inum` 改為 `dev == dev && inum == inum`（不考慮 refcount），讓已存在但無人引用的 inode 得以保留其記憶體中的 uid/gid。

```rust
// 修正前
if inode.r#ref > 0 && inode.dev == dev && inode.inum == inum {

// 修正後
if inode.dev == dev && inode.inum == inum {
```

### sys_chmod 交易記錄缺失

`sys_chmod` 缺少 `Operation::begin()`，導致 `inode.update()` 呼叫 `log::write()` 時 panic（`log_write: outside of trans`）。

## 已知限制

- 無權限檢查 — `sys_access` 只檢查路徑是否存在，不檢查權限位元
- `sys_setuid`/`sys_setgid` 無特權檢查（未實作 `EPERM`）
- uid/gid/mode 僅存於記憶體，未寫入磁碟（`DiskInode` 無對應欄位）
- 重開機後所有檔案權限重設為預設值
