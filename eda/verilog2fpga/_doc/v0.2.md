## ASC 格式解析 (`asc.rs`)

ASC 是 nextpnr 輸出的純文字配置檔，包含：

```
.io_tile 0 0        # io tile row=0, col=0
  .wiring 0 0 ...   # 繞線設定
.logic_tile 0 1     # logic tile row=0, col=1
  .lut 0 0 "1001"   # LUT 初始值 (16-bit)
  .ff 0 0 ce=1 sr=0 # Flip-Flop 屬性
  .carry 0 0 ci=... # 進位鏈
.connect row col net_id src dst...  # 連線矩陣
.synckey 0x12345678  # 同步金鑰
```

解析目標：
- 逐行掃描，區塊解析
- 每個 tile 轉為 `CramTile` 結構
- 支援 Logic Tile / IO Tile / BRAM Tile

## CRAM 記憶體模型 (`cram.rs`)

iCE40 CRAM 結構：

```
CRAM = [Frame; N_FRAMES]
Frame = [Word; N_WORDS_PER_FRAME]   // 每 frame = 33 words × 40 bits = 1320 bits
Word  = u40 (5 bytes)

iCE40HX8K:  70 rows × 200 cols → 每個 row 從左到右掃描 frames
iCE40HX1K:  30 rows × 120 cols
iCE40UP5K:  40 rows × 176 cols
```

- `cram.rs` 提供 `Cram::new(device)`
- `set_frame(row, col, data)` / `get_frame(row, col)`
- 含 CRC 計算與 Syndkey 寫入

## 位元流打包 (`pack.rs`)

流程：

```
ASC → parse() → Vec<CramTile> → layout() → Cram → serialize() → [u8]
```

序列化格式（iCE40 bitstream）：

```
[0x00] * 32          # 前導 32 個空白
[BitCount: u32 LE]   # 總位元數 (不含前導)
[Frame0][Frame1]...  # 每 frame = 33 words × 5 bytes = 165 bytes
[CRC32: u32 LE]      # CRC 校驗
```

- 空白前導：提供時間讓 FPGA 準備接收
- BitCount：payload 位元組數
- Frame 順序：依據 `v2f-db` 中的位址映射
- CRC：使用 iCE40 標準 CRC-32 (Ethernet)

## 驗收標準

1. `cargo build` 通過
2. 使用 nextpnr 產出一個 ASC 檔：
   - `v2f pack pure output.asc -o output.bin` (純 Rust)
   - `v2f pack icepack output.asc -o output_ref.bin` (呼叫 icepack)
   - 兩者比對 binary identical
3. 支援 HX1K / HX8K / UP5K 三種裝置
4. 所有 `v0.1` 功能仍正常運作
